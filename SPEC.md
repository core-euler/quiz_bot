ТЗ: Доработка backend Telegram-бота (интеграция с PlanDriver)
Общая идея
Дорабатывается существующий backend Telegram-бота.
Создается новый модуль интеграции с системой PlanDriver, который:
получает задания на тестирование водителей
запускает тесты в боте
отправляет результаты обратно в PlanDriver

1. Текущее состояние
Есть работающий Telegram-бот:
регистрация водителей
прохождение тестов
логика попыток и ограничений
хранение данных (Google Sheets / внутренняя логика)
админские функции
⚠️ Важно: текущую логику тестов не ломаем и не переписываем.

2. Что нужно сделать
Добавить в backend новый модуль:
👉 PlanDriver Integration Module
Который будет:
получать нарушения через API PlanDriver
превращать их в задания на тестирование
передавать задания в текущую систему тестов
после прохождения теста отправлять результат обратно

3. Архитектура
Общая схема
PlanDriver API ↑ │ polling ↓ PlanDriver Module (новый) ↓ Core логика бота (уже есть) ↓ Telegram 

4. Новый модуль PlanDriver
Создать отдельный модуль внутри backend, например:
services/plandriver/ 
Внутри:
plandriver_client.py # работа с API PlanDriver plandriver_sync.py # polling логика plandriver_mapper.py # преобразование данных plandriver_result_sender.py # отправка результатов 

5. Получение заданий (polling)
Реализовать фоновую задачу:
интервал: 1–5 минут
запрос:
GET /api/bot/pending-tests 
Обработка ответа
Для каждого водителя:
взять driver_id
взять attestation_id
пройти по violations
Для каждого нарушения:
violation_id
violation_type_code
comment

6. Создание задания в боте
Для каждого нового нарушения:
проверить, не обрабатывалось ли оно ранее
найти пользователя в Telegram
определить тест по violation_type_code
запустить тест через текущую систему

7. Маппинг тестов
Создать соответствие:
violation_type_code → тест 
Пример:
LATE → test_late TRAFFIC_VIOLATION → test_traffic VEHICLE_DAMAGE → test_vehicle 
Хранить в конфиге или таблице.

8. Маппинг водителей
Нужно связать:
driver_id ↔ telegram_id 
Создать таблицу:
driver_mapping 
Поля:
driver_id
telegram_id
personnel_number
full_name

9. Защита от дублей
Создать таблицу:
external_violations 
Поля:
violation_id
driver_id
attestation_id
status (new / sent / completed)
created_at
Используется для:
исключения повторной отправки
отслеживания статуса

10. Завершение теста
После прохождения теста:
определить, что тест был назначен через PlanDriver
собрать данные:
driver_id attestation_id violation_id violation_type_code passed score completed_at 
отправить:
POST /api/bot/test-result 

11. Логика результатов
если passed = true → нарушение закрывается
если passed = false → остается активным
повторные отправки допустимы (endpoint идемпотентен)

12. Обработка ошибок
Обработать:
водитель не найден
водитель не зарегистрирован
API PlanDriver недоступен
duplicate violation
timeout
ошибка отправки результата
Все ошибки логировать.

13. Ограничения
Не требуется:
менять текущую систему тестов
менять UX бота
обрабатывать критические нарушения
делать webhook

14. Требования к реализации
использовать текущий backend проекта
реализовать модульно (не смешивать с Telegram-логикой)
использовать Bearer авторизацию
реализовать polling worker
обеспечить идемпотентность
логировать все действия

15. Результат
После реализации:
нарушения автоматически попадают в бот
водители получают тесты без участия админов
результаты автоматически уходят в PlanDriver
нарушения закрываются автоматически

16. Ключевой принцип
👉 Интеграция не должна ломать существующую логику бота
Вся работа с PlanDriver — отдельный модуль.
