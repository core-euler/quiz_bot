# Интеграция Telegram-бота тестирования с PlanDriver

> Актуализировано по живому API `https://prog.lagrangegroup.ru`
> на `2026-04-01`.

## Общая схема

PlanDriver — система управления водительским составом. Бот тестирования — внешний сервис, который опрашивает API PlanDriver, получает задания на тестирование водителей и возвращает результаты. PlanDriver ничего не отправляет в бота — только отвечает на запросы.

**Base URL:** `https://prog.lagrangegroup.ru`
**OpenAPI документация:** https://prog.lagrangegroup.ru/docs
**Авторизация:** `Authorization: Bearer {token}` (Laravel Sanctum)

---

## Бизнес-логика

Водители получают нарушения. Нарушения делятся на **критические** и **некритические**.

### Некритические нарушения

```
Нарушение зафиксировано
  → Водитель ставится на контроль, создаётся аттестация (pending, дедлайн 7 дней)
  → Бот при polling видит задание → отправляет водителю тест по типу нарушения
  → Водитель проходит тест:
      Пройден → нарушение удаляется
        → Нарушений не осталось → аттестация passed, водитель снят с контроля
        → Ещё есть нарушения → ждём прохождения остальных тестов
      Не пройден → можно пересдать до дедлайна
  → Дедлайн истёк, тест не пройден:
      → Водитель в "список на разбор" (reason: test_not_passed)
```

### Критические нарушения

Критические типы в живом API:

- `4` — Отказ от рейса
- `6` — Повторные поломки
- `8` — Конфликт
- `9` — ДТП по вине водителя
- `10` — Недостаточный уровень навыков управления ТС

```
Нарушение зафиксировано
  → Водитель сразу попадает в "список на разбор" (reason: critical_violation)
  → Разбор в комиссии (дедлайн 14 дней):
      Прошёл → водитель остаётся
      Не прошёл или дедлайн истёк → увольнение
```

Важно:

- старая версия документации утверждала, что критические нарушения
  не попадают в `GET /api/bot/pending-tests`
- живая проверка `2026-04-01` показала, что в ответе
  `GET /api/bot/pending-tests` встречаются критические коды `4`, `6`, `9`

Следствие:

- нельзя опираться на старое утверждение как на контракт
- критические коды нужно обрабатывать отдельно на стороне бота
  и не запускать по ним обычный онлайн-тест

## API для бота

### GET /api/bot/pending-tests

Возвращает водителей с pending-аттестациями и нарушениями.

По живому API в объекте нарушения сейчас также приходит поле `is_critical`.

**Query-параметры:**
- `driver_id` (int, опционально) — фильтр по конкретному водителю

**Актуальный пример ответа:**

```json
{
  "data": [
    {
      "driver_id": 878,
      "driver_name": "Чижов Владимир Анатольевич",
      "personnel_number": "AUTO00131",
      "column_name": "АК Лагранж МСК 1 НП",
      "attestation_id": 400,
      "deadline": "2026-04-01",
      "violations": [
        {
          "violation_id": 641,
          "violation_type_code": "14",
          "violation_type_name": "Превышение нормы расхода топлива",
          "is_critical": false,
          "event_date": "2026-03-15",
          "comment": "ТС: О020АТ550 FAWJ7 FAWJ7. 41 л"
        }
      ]
    }
  ]
}
```

Если заданий нет — `"data": []`.

---

### POST /api/bot/test-result

Принимает результаты тестирования. Пройденные нарушения удаляются из системы. Если у водителя не осталось нарушений — аттестация автоматически закрывается как пройденная.

**Запрос:**

```json
{
  "driver_id": 878,
  "attestation_id": 400,
  "results": [
    {
      "violation_id": 641,
      "violation_type_code": "14",
      "passed": true,
      "score": 85,
      "completed_at": "2026-04-01T09:30:00Z"
    }
  ],
  "all_passed": true
}
```

| Поле | Тип | Обязательно | Описание |
|------|-----|-------------|----------|
| `driver_id` | int | да | ID водителя |
| `attestation_id` | int | да | ID аттестации |
| `results` | array | да | Массив результатов |
| `results[].violation_id` | int | да | ID нарушения |
| `results[].violation_type_code` | string | да | Код типа нарушения |
| `results[].passed` | bool | да | Тест пройден |
| `results[].score` | int | нет | Балл (0-100) |
| `results[].completed_at` | string | нет | ISO 8601 дата/время |
| `all_passed` | bool | да | Все тесты пройдены |

**Ответ:**

```json
{
  "status": "ok",
  "violations_cleared": [101],
  "violations_remaining": [98],
  "attestation_status": "pending",
  "message": "1 тест(ов) принято, осталось нарушений: 1"
}
```

Когда все нарушения закрыты:

```json
{
  "status": "ok",
  "violations_cleared": [101, 98],
  "violations_remaining": [],
  "attestation_status": "passed",
  "message": "Все тесты пройдены, аттестация завершена"
}
```

Эндпоинт идемпотентен — повторная отправка того же результата не вызывает ошибку.

## Точный Маппинг Для Бота

Согласованный с заказчиком онлайн-мэппинг относится только
к `9` некритическим нарушениям.

### Тематические блоки

| violation_type_code | Нарушение | Блок | Категория в Google Sheets |
|---------------------|-----------|------|----------------------------|
| `1` | Отклонение от маршрута | Логистика | `Логистика` |
| `2` | Простой без причины | Логистика | `Логистика` |
| `3` | Не отвечает на звонки | Логистика | `Логистика` |
| `5` | Проблемы по колёсам | Техническая часть | `Техническая часть` |
| `7` | Нарушение графика вахты | Правила компании | `Правила компании` |
| `11` | Несдача ТН | Правила компании | `Правила компании` |
| `12` | Нарушение скорости | Правила компании | `Правила компании` |
| `13` | Плохая работа с документацией | Работа с документацией | `Работа с документацией` |
| `14` | Превышение нормы расхода топлива | Экономия топлива | `Экономия топлива` |

### Критические коды

Коды `4`, `6`, `8`, `9`, `10` не должны запускать обычный онлайн-тест.

Даже если они приходят в `pending-tests`, для них нужен отдельный сценарий:

- либо пропуск на стороне бота
- либо отдельный workflow очного разбора

### Как задавать категории в Google Sheets

В листе `❓Вопросы`:

- используется колонка `категория`
- для вопросов PlanDriver надо использовать ровно эти канонические названия:
  - `Логистика`
  - `Техническая часть`
  - `Правила компании`
  - `Работа с документацией`
  - `Экономия топлива`

Практическое правило:

- в одной строке вопроса в колонке `категория` должно быть одно из этих значений
- не нужно писать коды `1`, `11`, `14`
- не нужно писать составные значения вроде
  `Логистика / Правила компании`

Пример:

| категория | вопрос |
|-----------|--------|
| `Логистика` | В каких случаях водитель обязан согласовать отклонение от маршрута? |
| `Правила компании` | В какие сроки водитель должен сдать ТН? |
| `Экономия топлива` | Что влияет на перерасход топлива в рейсе? |

Фактическая фильтрация в коде нечувствительна к регистру и лишним пробелам,
но в таблице лучше использовать именно канонические названия выше.

## Сценарий работы бота

```
1. Бот запрашивает GET /api/bot/pending-tests
2. Для каждого задания находит водителя в Telegram
   (маппинг driver_id ↔ Telegram ведётся на стороне бота)
3. Для некритических кодов отправляет водителю тест
   по `violation_type_code`
4. Для критических кодов не запускает обычный онлайн-тест
4. Водитель проходит тест → бот отправляет POST /api/bot/test-result
5. Повтор
```

Рекомендуемый интервал polling: 1-5 минут.

## Модель данных (справочно)

### Driver

| Поле | Тип | Описание |
|------|-----|----------|
| `id` | int | PK |
| `full_name` | string | ФИО |
| `personnel_number` | string | Табельный номер (уникальный) |
| `column_id` | int | Колонна (подразделение) |
| `status` | string | `working` / `fired` |

Модель Driver не содержит `telegram_id` — привязка к Telegram на стороне бота.

### Violation

| Поле | Тип | Описание |
|------|-----|----------|
| `id` | int | PK |
| `driver_id` | int | FK → Driver |
| `violation_type_id` | int | FK → ViolationType |
| `event_date` | date | Дата нарушения |
| `comment` | text | Комментарий |
