# PlanDriver Runbook

Этот документ нужен для живой проверки интеграции бота с PlanDriver.

## Что Уже Реализовано

- polling `GET /api/bot/pending-tests`
- локальное хранение `driver_mapping`
- локальное хранение `external_violations`
- защита от дублей по `violation_id`
- запуск теста из назначенного PlanDriver-сообщения
- отправка результата в `POST /api/bot/test-result`

## Что Нужно Перед Стартом

### 1. Env

Заполнить:

```env
PLANDRIVER_ENABLED=true
PLANDRIVER_BASE_URL=https://prog.lagrangegroup.ru
PLANDRIVER_TOKEN=...
PLANDRIVER_POLL_INTERVAL_MINUTES=5
PLANDRIVER_DB_PATH=/absolute/path/to/plandriver.db
PLANDRIVER_TEST_MAPPING='{}'
```

Примечание:
- `PLANDRIVER_TEST_MAPPING='{}'` подходит для первичной проверки доступа к backend.
- Для реального старта тестов нужен валидный маппинг категорий.

### 2. Google Sheets

Должны быть:

- подтверждённые пользователи в листе `👩‍👧‍👧Пользователи`
- валидные настройки в `⚙️Настройки`
- вопросы в `❓Вопросы`

Желательно:

- колонка `табельный номер` или `personnel_number`, если она уже есть в текущей таблице

### 3. Test Data

Нужно иметь хотя бы одного тестового водителя, для которого:

- в PlanDriver есть pending-аттестация
- в Google Sheets есть подтверждённый пользователь
- ФИО совпадает с `driver_name` из PlanDriver

Или:

- совпадает `personnel_number`, если такая колонка есть в листе пользователей

## Первичная Проверка Пути До Backend

Цель:
- убедиться, что бот вообще может ходить в backend PlanDriver

Шаги:

1. Запустить Redis
2. Запустить бота
3. Дождаться первого цикла polling
4. Посмотреть логи

Ожидаемые сигналы в логах:

- бот стартовал
- scheduler стартовал
- job `PlanDriver Sync` выполняется
- нет HTTP/URL ошибок до `prog.lagrangegroup.ru`

Если pending-назначений нет:
- это тоже валидный результат
- в логах должно быть `PlanDriver sync: no pending tests`

## Проверка Pending Assignments

Цель:
- увидеть реальные данные от `GET /api/bot/pending-tests`

Что нужно проверить:

- приходит `driver_id`
- приходит `driver_name`
- приходит `personnel_number`
- приходит `attestation_id`
- внутри `violations` есть `violation_id`, `violation_type_code`, `comment`

Что должно происходить локально:

- для нового нарушения создаётся запись в `external_violations`
- если водитель найден, статус становится `sent`
- если водитель не найден, статус остаётся `new` и пишется `last_error`

## Проверка Маппинга Водителя

Приоритет:

1. существующая запись в `driver_mapping`
2. совпадение по `personnel_number`, если такая колонка есть в Google Sheets
3. точное совпадение по ФИО

Что считать успешным результатом:

- бот находит ровно одного пользователя
- создаётся или обновляется `driver_mapping`

Что считать проблемой:

- не найден пользователь
- найдено несколько пользователей с одинаковым ФИО

## Проверка Доставки Назначения

После успешного маппинга бот должен:

- отправить отдельное сообщение в Telegram
- приложить кнопку `Начать тест`

В сообщении должны быть:

- тип нарушения
- комментарий
- дедлайн

## Проверка Старта Теста

После нажатия на кнопку:

- бот должен запустить текущую систему тестов
- вопросы должны выбираться из категорий, указанных в `PLANDRIVER_TEST_MAPPING`

Если маппинг или категории не готовы:

- запуск теста может упереться в отсутствие подходящих вопросов

## Проверка Отправки Результата

После завершения теста бот должен:

- записать результат в Google Sheets
- определить, что тест пришёл из PlanDriver
- отправить `POST /api/bot/test-result`

Что проверить:

- `driver_id`
- `attestation_id`
- `violation_id`
- `violation_type_code`
- `passed`
- `score`
- `completed_at`
- `all_passed`

## Проверка При `passed=false`

Ожидаемое поведение:

- результат уходит в PlanDriver
- локальный статус нарушения остаётся `sent`
- нарушение не считается закрытым

## Проверка Ошибок

Нужно отдельно проверить:

- водитель не найден
- водитель не зарегистрирован в боте
- API PlanDriver недоступен
- duplicate violation
- timeout на вопрос
- ошибка отправки результата

## Что Считать Готовностью

Интеграцию можно считать реально проверенной, когда подтверждены все пункты:

- бот успешно ходит в `GET /api/bot/pending-tests`
- бот находит тестового водителя
- бот отправляет назначение в Telegram
- тест стартует
- тест завершается
- бот отправляет `POST /api/bot/test-result`
- при `passed=true` нарушение локально становится `completed`
- при `passed=false` нарушение остаётся активным
