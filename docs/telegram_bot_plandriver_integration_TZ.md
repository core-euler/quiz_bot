## Доработка Telegram‑бота тестирования для интеграции с системой PlanDriver

**Проект:** Telegram‑бот для тестирования водителей\
**Цель доработки:** интеграция с системой PlanDriver для автоматического
назначения тестов по нарушениям и передачи результатов тестирования.

------------------------------------------------------------------------

# 1. Текущее состояние системы

На данный момент уже реализован и работает Telegram‑бот, который:

-   регистрирует водителей (ФИО, колонна, телефон)
-   проводит тестирование
-   хранит результаты тестов
-   имеет административные функции
-   использует Google Sheets как хранилище данных
-   поддерживает повторное тестирование

Доработка не предполагает переписывание бота.

Требуется добавить **backend‑прослойку на FastAPI**, которая будет:

1.  взаимодействовать с API системы PlanDriver
2.  получать задания на тестирование
3.  отправлять тесты водителям через существующего Telegram‑бота
4.  передавать результаты тестирования обратно в PlanDriver

------------------------------------------------------------------------

# 2. Архитектура интеграции

## Общая схема

PlanDriver ничего не отправляет в бот.

Бот **сам опрашивает API PlanDriver**.

    FastAPI backend (бот)
            ↓ polling
    GET /api/bot/pending-tests
            ↓
    PlanDriver API
            ↓
    получаем задания
            ↓
    отправляем тест водителю
            ↓
    водитель проходит тест
            ↓
    POST /api/bot/test-result

# 5. Получение заданий на тестирование

Endpoint

    GET /api/bot/pending-tests

Описание

Возвращает список водителей, у которых есть нарушения и назначенные
тесты.

Пример ответа

``` json
{
  "data": [
    {
      "driver_id": 42,
      "driver_name": "Иванов Иван Иванович",
      "personnel_number": "12345",
      "column_name": "Колонна №3",
      "attestation_id": 15,
      "deadline": "2026-03-20",
      "violations": [
        {
          "violation_id": 101,
          "violation_type_code": "LATE",
          "violation_type_name": "Опоздание",
          "event_date": "2026-03-10",
          "comment": "Опоздание на 30 минут"
        }
      ]
    }
  ]
}
```

Каждое нарушение соответствует отдельному тесту.

------------------------------------------------------------------------

# 6. Отправка результатов тестирования

Endpoint

    POST /api/bot/test-result

Payload

``` json
{
  "driver_id": 42,
  "attestation_id": 15,
  "results": [
    {
      "violation_id": 101,
      "violation_type_code": "LATE",
      "passed": true,
      "score": 85,
      "completed_at": "2026-03-15T14:30:00Z"
    }
  ],
  "all_passed": true
}
```

Поведение:

-   если `passed = true` → нарушение удаляется
-   если нарушений больше нет → аттестация закрывается

Endpoint **идемпотентен**.

------------------------------------------------------------------------

# 7. Типы нарушений

Некритические нарушения обрабатываются ботом.

Критические нарушения не обрабатываются.

  code                name                         critical
  ------------------- ---------------------------- ----------
  LATE                Опоздание                    нет
  ABSENCE             Неявка                       нет
  TRAFFIC_VIOLATION   Нарушение ПДД                нет
  VEHICLE_DAMAGE      Повреждение ТС               нет
  CARGO_DAMAGE        Повреждение груза            нет
  DOCUMENTATION       Нарушение документооборота   нет
  OTHER               Прочее                       нет

Критические:

-   DRUNK
-   INSUBORDINATION

# 8. Логика работы FastAPI backend

## Polling worker

Каждые **1--5 минут**:

1.  отправляет запрос

```{=html}
<!-- -->
```
    GET /api/bot/pending-tests

2.  получает список нарушений
3.  проверяет новые нарушения
4.  отправляет тест водителю через Telegram

------------------------------------------------------------------------

# 9. Маппинг тестов

Каждый тип нарушения соответствует тесту.

Пример:

    LATE → test_late
    TRAFFIC_VIOLATION → test_traffic
    VEHICLE_DAMAGE → test_vehicle
    CARGO_DAMAGE → test_cargo

Маппинг хранится в конфигурации backend.

------------------------------------------------------------------------

# 10. Связь PlanDriver и Telegram

PlanDriver не хранит Telegram ID.

Необходимо хранить таблицу маппинга.

## Таблица driver_mapping

  поле               описание
  ------------------ --------------------------
  driver_id          ID из PlanDriver
  telegram_id        ID пользователя Telegram
  personnel_number   табельный номер
  full_name          ФИО

Маппинг создаётся при регистрации водителя в боте.

------------------------------------------------------------------------

# 11. Предотвращение повторной отправки тестов

Нужно хранить отправленные задания.

## Таблица sent_tests

  поле           описание
  -------------- ---------------------------
  violation_id   ID нарушения
  driver_id      ID водителя
  sent_at        дата отправки
  status         pending / passed / failed

Это необходимо, чтобы polling не отправлял один и тот же тест повторно.

# 13. Алгоритм работы системы

    1. cron запускает polling
    2. FastAPI вызывает GET /api/bot/pending-tests
    3. получает список нарушений
    4. ищет telegram_id по driver_id
    5. отправляет тест водителю
    6. водитель проходит тест
    7. бот отправляет POST /api/bot/test-result
    8. PlanDriver закрывает нарушение

------------------------------------------------------------------------

# 14. Обработка ошибок

Система должна корректно обрабатывать:

-   водитель не найден в Telegram
-   водитель не зарегистрирован
-   ошибка API PlanDriver
-   повторная отправка результатов
-   истечение дедлайна

Все ошибки должны логироваться.

------------------------------------------------------------------------

# 15. Требования к реализации

Backend должен:

-   быть реализован на **FastAPI**
-   поддерживать **Bearer авторизацию**
-   иметь **polling worker**
-   использовать **логирование**
-   иметь **идемпотентную отправку результатов**
-   корректно обрабатывать повторные задания

------------------------------------------------------------------------

# 16. Результат работы

После реализации:

-   PlanDriver автоматически назначает тесты
-   бот получает задания через API
-   водители проходят тесты
-   результаты автоматически возвращаются в систему
-   нарушения закрываются без участия операторов
